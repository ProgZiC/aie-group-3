## 2. Постановка задачи и контекст

Опишите:

1. **Предметную область и задачу:**
   - Решается задача бинарной классификации табличных данных: выявление клиентов, склонных к оттоку.
   - Потенциальные пользователи: менеджеры по удержанию клиентов (Retention), маркетологи, аналитики. Сценарий: система анализирует абонентов, выявляет лиц с высоким риском ухода и формирует списки для предложения персональных скидок или бонусов.
2. **Формулировку задачи в терминах ML/ИИ:**
   - вектор характеристик клиента (социально-демографические данные, подключенные услуги, тип контракта, суммы платежей).
   - Churn (отток). Сервис возвращает бинарную метку (1 — уйдет, 0 — останется) и точную вероятность ухода (probability).
   - высокая скорость ответа API для интеграции с веб-витринами, необходимость интерпретируемости предсказаний (бизнесу важно понимать причину ухода).

3. **Целевые метрики качества:**
   - F1 (выбрана из-за сильного дисбаланса классов в датасете,ушедших клиентов значительно меньше, и нам важен баланс между точностью их выявления и полнотой), ROC-AUC (основная метрика для оценки качества ранжирования вероятностей оттока.)

## 4. Модели и подходы

Опишите, какие модели вы пробовали и как развивался ваш подход:

1. **Базовые (baseline) модели:**
   - В качестве baseline использовалась логистическая регрессия (LogisticRegression). Перед подачей данных применялось масштабирование числовых признаков (StandardScaler) и One-Hot Encoding для категорий.
   - Базовая модель сразу задала высокую планку, показав отличный F1-Score (0.6088) и ROC-AUC (0.8361).

2. **Улучшенные модели и эксперименты:**
   - Использован градиентный бустинг CatBoostClassifier. Промежуточный эксперимент: Модель была обучена "из коробки" (с базовыми параметрами). Наблюдалась просадка по метрике F1-Score (0.5663), что является типичным поведением бустинга на несбалансированных данных без применения весов классов (модель слишком часто предсказывает мажоритарный класс).
   - Для повышения метрик использована байесовская оптимизация гиперпараметров через Optuna.
   - Для решения проблемы "черного ящика" бустинга и обеспечения интерпретируемости применена библиотека SHAP (Summary и Waterfall plot).

3. **Нейросетевые модели (если применимо):**
   - Не применялись, так как градиентный бустинг (CatBoost/XGBoost) является стандартом де-факто (SOTA) для подобных табличных задач без потери интерпретируемости.
   

Ссылайтесь на конкретные ноутбуки с экспериментами (например, `notebooks/02_baselines.ipynb`, `notebooks/03_models_tuning.ipynb`).
- Шаги обучения зафиксированы в `notebooks/02_models.ipynb`.

## 5. Экспериментальный протокол и результаты

Здесь важно показать, что выбор финальной модели был осмысленным.

1. **Экспериментальный протокол:**
   - Данные разделены на обучающую (train) и тестовую (test) выборки в пропорции 80/20. Зафиксирован seed (random_state=42) для воспроизводимости.
   - Использовалась стратификация по целевой переменной (stratify=y) для сохранения дисбаланса классов в обеих частях выборки.

2. **Сравнение моделей по метрикам:**

   Пример таблицы (адаптируйте под свои метрики):

   | Модель / конфигурация         | Описание                                    |  Метрика 1  |  Метрика 2  |        Комментарий                |
   |-------------------------------|---------------------------------------------|-------------|-------------|-----------------------------------|
   | Baseline (LogReg)                    | LogisticRegression, стандартные фичи | 0.8361      | 0.6088      | Простая стартовая точка           |
   | CatBoost (Default)                   | CatBoost, базовые параметры          | 0.8358      | 0.5663      | Просадка F1 из-за дисбаланса      |
   | CatBoost + Optuna                    | CatBoost, тюнинг                     | 0.8386      | 0.5765      | Лучший ROC-AUC, финальная модель  |

3. **Выбор финальной модели:**
   - Несмотря на то, что логистическая регрессия показала более высокий F1-Score, финальной моделью для сервиса выбрана CatBoost + Optuna.
   - Обоснование: CatBoost продемонстрировал наилучшее значение ROC-AUC (0.8386), что означает его превосходство в ранжировании клиентов по риску оттока (от наиболее вероятных к наименее). В бизнес-сценариях удержания именно качество вероятностей (predict_proba) зачастую важнее жестких классов. Кроме того, использование градиентного бустинга в связке с SHAP позволяет "извлекать" понятные бизнесу причины ухода для каждого конкретного клиента, что делает эту архитектуру более предпочтительной для комерческого использования.

## 6. Архитектура решения и сервис

Опишите, как из модели получился работающий сервис:

1. **Архитектура пайплайна:**
   - REST API получает JSON -> валидация Pydantic -> конвертация в pandas DataFrame -> выравнивание признаков (заполнение недостающих категорий нулями) -> нормализация через `scaler.pkl` -> предсказание моделью `catboost_churn_model.cbm` -> выдача JSON-ответа.

2. **API и endpoints:**
   - GET `/health` — проверка состояния сервиса (отдает {"status": "ok"}) и флага успешной загрузки модели.
   - POST `/predict` — принимает JSON с 20 параметрами абонента (по дефолту подставляется тестовый клиент) и возвращает ответ формата: {"prediction": 1, "probability": 0.7134, "model_version": "v1.0"}.

3. **Технологический стек:**
   - ML: pandas, scikit-learn, catboost, optuna, shap.
   - Web: FastAPI, uvicorn, pydantic.
   - Инфраструктура и развертывание: локальное виртуальное окружение. Для удобного поднятия используются написанные скрипты автоматизации `start.sh` (для Linux/macOS) и `start.bat` (для Windows), которые самостоятельно устанавливают зависимости из `requirements.txt` и запускают сервер.
